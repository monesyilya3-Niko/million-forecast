# 百万竞彩稳定性修复与终端 UI 重构设计

## 目标

将现有 Claude 生成的 Streamlit 项目修复为可启动、可导航、可导入数据、可在空数据环境运行的本地分析终端，并在保留足球、排列三和大乐透业务能力的前提下，建立统一的金融情报终端视觉系统。

产品名称统一为“百万竞彩 / Million Forecast Terminal”。系统继续使用 Python 3.13、Streamlit、DuckDB、Plotly、Pandas、NumPy 和 scikit-learn，不引入外部前端框架、CDN 或 JavaScript。

## 当前基线

- Python 3.13 可编译 `app.py` 与 `src` 下共 81 个 Python 文件。
- Ruff 检查通过。
- 现有 22 项 pytest 测试通过。
- 现有测试集中于足球模型与服务，未覆盖彩票模块。
- 当前结果不能证明彩票校验、空数据库、全部页面、导入、导航返回和 Streamlit 实际启动均可靠。
- 项目此前不是 Git 仓库；本次仅初始化本地仓库和本地提交身份。

## 设计原则

1. 先证明故障，再修复根因；每个行为变更先建立失败测试。
2. 先修运行与数据正确性，再调整视觉层。
3. 保留成熟足球模型和服务接口，避免无关重写。
4. 空数据、缺少数据源和未实现模块都是正常状态，页面不得崩溃。
5. 不伪造实时数据、奖金、收益或模型能力。
6. 所有足球和彩分析输出均明确概率性质、数据限制和风险。
7. 正式 DuckDB 结构变更前创建本地备份，并采用向后兼容迁移。

## 架构

保留现有分层：

- `app.py`：应用启动、依赖组装、页面注册与全局导航。
- `ui/pages`：页面编排，不直接承载复杂业务规则。
- `ui/components.py`：设计令牌、HTML 安全处理、通用组件和 Plotly 主题。
- `services`：足球分析、训练、实时状态和数据质量用例。
- `lottery`：彩票模型、校验、仓储、生成与回测逻辑。
- `data`：DuckDB 初始化、迁移、仓储和外部数据适配。

页面由统一注册表描述分组、标题、渲染函数和状态。注册函数缺失或模块未实现时，应用显示标准空状态并记录错误，不因单页问题导致整个应用不可用。

## 数据与错误处理

### 导入流程

CSV 和 JSON 导入统一经过：

1. 文件类型和编码解析。
2. 必需字段与字段类型检查。
3. 排列三或大乐透业务规则校验。
4. 重复期号与现有数据冲突检查。
5. 在事务中批量写入。
6. 返回成功、跳过和失败行的明确统计。

任何解析或校验失败均不得产生部分脏写入。错误反馈包含行号和原因，但不暴露内部堆栈。

### 空数据与安全访问

- DataFrame 渲染和列访问前检查空值及必需列。
- `None`、`NaN`、无穷值和缺失字段通过统一格式化函数处理。
- 查询无结果时返回稳定的空结构，不要求调用方猜测字段。
- 外部数据源不可用时显示数据源状态和可执行的恢复提示。

### 数据库

- 所有表通过 `CREATE TABLE IF NOT EXISTS` 初始化。
- 迁移逻辑可重复执行。
- 彩票期号约束明确彩种作用域；若继续分表，则分别保证期号唯一，统一预测和回测表使用彩种与期号的组合约束。
- 日期使用可排序类型，JSON 数据采用一致的序列化规则。
- 正式数据库迁移前创建带时间戳的本地备份。

## 导航与 Streamlit 状态

- 统一管理当前页、来源页、页面历史和筛选快照。
- 从今日竞彩、价值发现、排列三和大乐透页面进入详情后，返回时恢复来源页及筛选条件。
- 页面刷新不无故回到默认首页。
- 每个交互组件使用稳定且唯一的显式 key。
- `st.set_page_config` 仅由入口调用一次。
- 对当前 Streamlit 版本不支持的参数使用兼容实现。

## 彩票模型与回测

### 校验

- 排列三：恰好三位、每位 0–9，并验证和值、跨度、形态、奇偶比、大小比、012 路、重号和连号。
- 大乐透：前区五个不重复号码 1–35，后区两个不重复号码 1–12；统一排序并验证和值、跨度、奇偶比、大小比、分区比、连号和重号。

### 生成与模型

- 随机基线支持显式 seed，以便复现。
- 空样本、样本不足和除零条件返回明确状态。
- 大乐透组合生成设置有界尝试次数，禁止无限循环和重复组合。
- 页面使用“候选组合”“参考组合”“组合筛选结果”等中性表达。

### 回测

- 每一期只能使用该期之前的数据，禁止未来信息泄漏。
- 策略与随机基线使用相同评估窗口。
- 没有真实奖金数据时不计算或编造 ROI，改为报告覆盖率、命中分布和明确限制。

## 视觉系统

采用已批准的 A 方向“金融情报终端”：

- 深海军蓝分层背景与克制径向光晕，不使用纯黑。
- 青蓝作为品牌与信息色；绿色、黄色和红色仅表达状态语义。
- 低透明度玻璃卡片、细边框、高密度栅格和清晰层级。
- Sidebar 包含品牌、英文名、模块范围、本地运行状态、数据库状态、数据源状态和最近同步时间。
- 页面统一结构为 eyebrow、hero、核心指标、主分析区、数据质量和风险声明。
- 足球赔率采用紧凑交易终端表达，展示市场时间戳、变化方向与质量状态。
- 彩票号码球使用细描边、内高光和受控的前后区配色。
- Plotly 使用透明背景、统一坐标轴、网格、图例、悬浮信息和颜色序列。
- CSS 集中在组件层，页面不重复注入主题样式。

## 通用组件

组件层需要提供并测试：

- 安全与格式化：`safe_html`、`format_number`、`format_percent`。
- 框架：`apply_theme`、`plotly_theme`、`hero_pro`、`section_header`。
- 状态：`metric_card`、`render_badge`、`render_status_dot`、`render_confidence_meter`、`empty_state`。
- 风险与质量：`render_risk_note`、`risk_panel`、`data_quality_badge`、`data_quality_panel`、`provider_status_panel`。
- 足球：`match_card`、`odds_box`、`probability_triplet`、`tactical_card`。
- 彩票：`lottery_number_ball`、`lottery_number_row`、`lottery_draw_card`、`lottery_pattern_badge`、`lottery_metric_grid`、`lottery_risk_disclaimer`、`candidate_combo_card`、`backtest_summary_card`。
- 保留现有页面依赖的兼容函数，不在迁移完成前删除旧 API。

## 页面策略

优先完成现有注册页面的可靠性和高级 UI。需求中尚不存在的页面先注册为明确的“建设中”或“数据源未配置”空状态，不伪装成已实现功能。

页面升级顺序：

1. 今日竞彩、比赛分析、比赛详情、价值发现。
2. 排列三分析、历史、组合生成。
3. 大乐透分析、历史、组合生成。
4. 足球与彩票回测。
5. 数据中心、模型中心和系统状态。
6. 其余已有足球分析页与占位页。

## 测试与验收

### 自动化测试

- 彩票号码、衍生指标、重复期号、导入和组合生成测试。
- 空数据库和缺少字段测试。
- 数据库初始化及可重复迁移测试。
- 页面注册、导航状态和唯一 widget key 测试。
- 通用组件安全转义和异常值格式化测试。
- 回测时间边界、随机基线和无奖金数据测试。
- 所有页面使用临时空数据库的烟雾测试。

### 命令验证

- Python 编译检查。
- Ruff 全项目检查。
- pytest 全量测试。
- 临时数据库初始化和健康检查。
- `streamlit run` 启动探测。
- 浏览器验证关键页面、空状态、导航返回和导入交互。

### 完成标准

- 应用和首页可启动。
- 所有注册页面可打开，空数据不报错。
- CSV/JSON 模板可下载，非法彩票数据被拒绝，合法数据可导入。
- 页面返回状态与筛选条件正确保留。
- 无明显 DuplicateWidgetID、KeyError、AttributeError 或 ImportError。
- 足球、彩票和回测页面使用统一终端设计与 Plotly 主题。
- 所有风险文案不承诺收益、确定赛果或中奖。
- 不破坏现有足球功能，不编造真实数据。

## 非目标

- 不迁移到 React、Vue 或其他前端框架。
- 不增加云部署、用户账户、计费或远程数据库。
- 不伪造缺失的实时阵容、伤停、赔率、开奖结果或奖金数据。
- 不以 UI 占位内容替代真实业务实现。
